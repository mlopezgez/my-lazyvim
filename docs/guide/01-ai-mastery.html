Introduction

This configuration includes claudecode.nvim, a powerful integration that brings Claude AI directly into your Neovim workflow. Think of Claude as an AI pair programmer‚Äînot a replacement for your skills, but a tool to amplify them.

When to Use Claude

  • Architecture & Planning: Brainstorm system designs, discuss trade-offs
  • Boilerplate Generation: Quickly scaffold repetitive code structures
  • Debugging: Get fresh perspectives on tricky bugs
  • Code Review: Ask Claude to review your changes for issues
  • Documentation: Generate docstrings, comments, and type annotations
  • Refactoring: Transform code with AI-suggested improvements
üí° Pro Tip

Claude works best when you provide context. The more relevant code you share, the better its suggestions will be. Use <leader>ab to add buffers before asking questions.

Core Workflows

Session Management

Claude runs in a terminal split within Neovim (30% of window width by default). You can toggle it on/off, resume previous conversations, and focus the terminal when needed.

Keybind Description When to Use
<leader>ac Toggle Claude terminal Open/close Claude split. Your primary command.
<leader>af Focus Claude terminal Jump cursor to Claude window without toggling.
<leader>ar Resume previous session Continue a past conversation with full context.
<leader>aC Continue last conversation Pick up where you left off in recent chat.

Context Management

The key to effective AI assistance is providing the right context. Claude can't see your entire codebase by default‚Äîyou need to explicitly share relevant files and code snippets.

Keybind Mode Description
<leader>ab normal Add current buffer to Claude's context
<leader>as visual Send visual selection to Claude
<leader>as normal Add file from file explorer (in explorer only)
‚ö°

Pattern 1: Multi-File Refactoring

  1. Identify all related files (e.g., a class and its tests)
  2. Open the first file: nvim src/user.py
  3. Add it to context: <leader>ab
  4. Open the test file: :e tests/test_user.py
  5. Add it to context: <leader>ab
  6. Toggle Claude: <leader>ac
  7. Ask: "Refactor the User class to use dataclasses. Update tests accordingly."
  8. Review proposed changes
  9. Accept or modify as needed
üí° Tip: Add up to 3-5 related files for best results. Too many files can dilute focus.

Model Selection

Claude offers different models with varying capabilities and speeds. Choose based on your task complexity.

Keybind Description
<leader>am Select Claude model (Sonnet/Opus)

Model Comparison

  • Sonnet 4.5 (Recommended): Fast, cost-effective, excellent for most tasks. Use for: quick questions, code generation, refactoring.
  • Opus 4.5: Most capable, slower, higher cost. Use for: complex architecture decisions, critical bug analysis, advanced system design.
üí° When to Switch Models

Start with Sonnet for 95% of tasks. Only switch to Opus when you need deep reasoning about complex systems or are stuck on a particularly challenging problem.

Diff Workflow

When Claude suggests code changes, it presents them as diffs. You can accept or deny these changes directly from Neovim.

Keybind Description When to Use
<leader>aa Accept Claude's diff changes Apply the suggested changes to your file.
<leader>ad Deny/reject Claude's diff Discard the suggestions and keep original code.
‚ö°

Pattern 2: Test-Driven Development with AI

  1. Write a failing test case first (TDD approach)
  2. Select the test with visual mode: V
  3. Send to Claude: <leader>as
  4. Ask: "Implement the function to pass this test"
  5. Claude proposes implementation
  6. Review the diff in the terminal
  7. If good, accept: <leader>aa
  8. Run your tests to verify
  9. Iterate if needed
üí° Best Practice: Always review diffs carefully. Claude is helpful but not infallible.

Advanced Techniques

Effective Prompting Strategies

The quality of Claude's responses depends heavily on how you ask questions. Here are proven strategies for getting the best results:

‚úÖ Good Prompts

  • "This function uses ty LSP with Poetry. Can you add complete type annotations?"
  • "Refactor this to follow the repository pattern. Here's an example from our codebase..."
  • "I'm getting a TypeError on line 42. The diagnostic says: [paste with <leader>cy]"

‚ùå Vague Prompts

  • "Make this better" (Better how? Performance? Readability?)
  • "Fix this" (What's broken? What error are you seeing?)
  • "Add features" (Which features? What's the use case?)
üí° Context is King

Before asking Claude to modify code, always mention:

  1. Your tooling: "This uses ty LSP" or "We're using FastAPI"
  2. Your patterns: "Follow the async/await pattern used elsewhere"
  3. Your constraints: "Must work with Python 3.10+"

Pattern 3: Documentation Generation

  1. Write your function/class implementation first

Pattern 4: Code Review Before Commit

  1. Make your changes
  2. Stage files: git add .
  3. View diff: git diff --cached
  4. Copy the diff output
  5. Toggle Claude: <leader>ac
  6. Paste and ask: "Review this diff for bugs, edge cases, and code quality issues"
  7. Address any issues Claude identifies
  8. Commit with confidence
‚ö°

Pattern 5: Error Resolution

  1. Encounter an error or failing test
  2. Navigate to the error line
  3. Copy diagnostic: <leader>cy
  4. Add affected buffer: <leader>ab
  5. Toggle Claude: <leader>ac
  6. Paste error and ask: "Here's the error: [paste]. How do I fix this?"
  7. Apply the suggested fix
  8. Verify with LSP or tests
üí° Pro Tip: The <leader>cy keybind automatically copies diagnostics to system clipboard!

Troubleshooting

Claude not responding?

  • Focus the terminal: <leader>af
  • Check if the process is running
  • Close and reopen: <leader>ac twice
  • Verify ~/.local/bin/claude exists

Wrong context in responses?

  • Start a fresh session (don't resume old ones)
  • Re-add the correct buffers: <leader>ab
  • Be explicit: "Ignore previous context. Focus on this file..."

Diffs too large or overwhelming?

  • Ask Claude to break changes into smaller steps
  • Use visual selection for specific sections
  • Request: "Only show changes for the parse_data function"
